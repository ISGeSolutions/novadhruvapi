# Code Generation Prompt: Nova.Shared + Nova.Shell.Api

## Instruction

Generate complete, production-quality code for three .NET 10 projects:
1. `Nova.Shared` — pure .NET shared infrastructure library (no ASP.NET Core dependency)
2. `Nova.Shared.Web` — ASP.NET Core extensions layer (middleware, JWT, web OTel instrumentation)
3. `Nova.Shell.Api` — thin shell API with 4 diagnostic endpoints (MSSQL, Postgres, MySQL, hello-world)

Do not add any code, abstractions, or features beyond what is explicitly specified.
Do not generate placeholder comments saying "add logic here" — implement it properly or omit it.

## Standard Audit Columns (All Tables)

Every database table has these columns, in this order, after domain columns:

```sql
frz_ind         bit / boolean           NOT NULL DEFAULT 0   -- soft delete flag
created_on      datetime2 / timestamptz NOT NULL
created_by      nvarchar(10) / varchar(10) NOT NULL
updated_on      datetime2 / timestamptz NULL
updated_by      nvarchar(10) / varchar(10) NULL
updated_at      nvarchar(45) / varchar(45) NULL              -- IP address (IPv4 or IPv6)
```

- `frz_ind = 1` means frozen/soft-deleted. All queries must include `WHERE frz_ind = 0` (or `false` on Postgres).
- `updated_at` stores the client IP address, not a datetime.
- Audit fields are populated in infrastructure from `TenantContext` and `IHttpContextAccessor`.

| Database       | PK column | PK type                  | Generated by |
|----------------|-----------|--------------------------|--------------|

- Infrastructure layer parses `string` to `int` (MSSQL/MySQL) or `Guid` (Postgres) based on `TenantContext.DbType`.

## Cross-Database / Cross-Schema Table References

| Database  | Notation                        | Example                          |
|-----------|---------------------------------|----------------------------------|
| MSSQL     | `database.schema.table`         | `sales97.dbo.pointer`            |
| Postgres  | `schema.table` (within one DB)  | `sales97.pointer`                |
| MySQL     | `schema.table` (within one DB)  | `sales97.pointer`                |

In Postgres and MySQL, each tenant has one database. MSSQL "databases" (e.g. `sales97`, `presets`) map to Postgres/MySQL **schemas** within that single tenant database.

`ISqlDialect` must provide a `TableRef(string databaseOrSchema, string table)` method:
- MSSQL: returns `databaseOrSchema + ".dbo." + table`
- Postgres: returns `databaseOrSchema + "." + table`
- MySQL: returns `databaseOrSchema + "." + table`

## Multi-Tenancy Model

- **One database per tenant** (MSSQL or Postgres — never mixed for a single tenant)
- Tenant is resolved from the **JWT claim** `tenant_id`
- `TenantContext` (scoped, per-request) carries:

```csharp
public sealed record TenantContext
{
    public string TenantId         { get; init; }
    public string ConnectionString { get; init; }    // already decrypted
    public DbType DbType           { get; init; }    // MsSql | Postgres | MySql
    public string SchemaVersion    { get; init; }    // "legacy" | "v1"
}

public enum DbType { MsSql, Postgres, MySql }
```

- `TenantRegistry` loads tenant records from configuration (see appsettings.json below)
- `TenantResolutionMiddleware` runs early in the pipeline, resolves `TenantContext` from JWT, registers it as a scoped service
- For the shell's `/test-db/*` endpoints, tenant resolution is bypassed — explicit connections are used from named config sections

## Program.cs Requirements

- Supports two modes determined by environment variable `RUN_AS_CONSOLE=true` or presence of `--console` in args:
  - **Web mode**: normal Kestrel/ASP.NET Core host
  - **Console mode**: same host but `UseConsoleLifetime()`, verbose startup output to stdout
- In console mode, print to stdout after each step:
  - Config files loaded and validated
  - MSSQL connectivity result (ping)
  - Postgres connectivity result (ping)
  - Tenant registry loaded (count of tenants)
  - OTel configured (endpoint printed)
- Wires up in this order:
  1. `AddNovaConfiguration(builder)` — loads both config files, validates OpsSettings
  2. `AddNovaLogging(builder)` — Serilog with file sinks
  3. `AddNovaTenancy(services)` — TenantRegistry, TenantContext DI
  4. `AddNovaOpenTelemetry(builder)` — base OTel: runtime metrics, OTLP exporter, ActivitySource
  5. `AddNovaWebInstrumentation(builder)` — ASP.NET Core traces + metrics instrumentation
  6. `AddNovaJwt(builder)` — JWT authentication
  7. Health checks (MSSQL + Postgres)
  8. Endpoint registration
- `CorrelationIdMiddleware` and `TenantResolutionMiddleware` added to pipeline (from `Nova.Shared.Web`)

## Endpoint Specifications

### `GET /hello-world`

- No auth required
- Returns JSON:
```json
{
  "message": "Hello, World!",
  "timestamp": "2026-03-13T10:00:00Z",
  "correlationId": "abc-123"
}
```
- `correlationId` taken from `IHttpContextAccessor` (set by `CorrelationIdMiddleware`)

### `GET /test-db/mssql`

- No tenant resolution — uses `DiagnosticConnections:MsSql` from `appsettings.json`
- Executes via Dapper:
```sql
SELECT code, value FROM sales97.dbo.pointer
```
- Uses `MsSqlDialect.TableRef("sales97", "pointer")` to build the table reference
- Returns JSON array:
```json
[{ "code": "ABC", "value": "Something" }]
```
- On connection failure: returns `503` with structured error body `{ "error": "...", "db": "mssql" }`

### `GET /test-db/postgres`

- No tenant resolution — uses `DiagnosticConnections:Postgres` from `appsettings.json`
- Executes via Dapper:
```sql
SELECT code, value FROM sales97.pointer
```
- Uses `PostgresDialect.TableRef("sales97", "pointer")` to build the table reference
- Returns JSON array: same shape as MSSQL endpoint
- On connection failure: returns `503` with structured error body `{ "error": "...", "db": "postgres" }`

### `GET /test-db/mysql`

- No tenant resolution — uses `DiagnosticConnections:MySql` from `appsettings.json`
- Executes via Dapper:
```sql
SELECT code, value FROM sales97.pointer
```
- Uses `MySqlDialect.TableRef("sales97", "pointer")` to build the table reference
- Returns JSON array: same shape as other test-db endpoints
- On connection failure: returns `503` with structured error body `{ "error": "...", "db": "mysql" }`

### `GET /health/mysql`

- Uses `MySqlHealthCheck` registered with ASP.NET Core health checks
- Executes `SELECT 1` against `DiagnosticConnections:MySql`

### `GET /health/mssql`

- Uses `MsSqlHealthCheck` registered with ASP.NET Core health checks
- Executes `SELECT 1` against `DiagnosticConnections:MsSql`
- Returns standard health check JSON response

### `GET /health/postgres`

- Uses `PostgresHealthCheck`
- Executes `SELECT 1` against `DiagnosticConnections:Postgres`
- Returns standard health check JSON response

### `GET /health`

- Aggregate health check combining MSSQL, Postgres, and MySQL checks

## Encryption / Decryption — CipherService

An existing `cipher.cs` file will be provided by the user at implementation time. The code generation must:

1. Include `cipher.cs` verbatim in `Nova.Shared/Security/` — do not modify it
2. Wrap it in `CipherService` as shown below — do not invent a different algorithm

```
Nova.Shared/
└── Security/
    ├── cipher.cs           -- provided as-is, do not alter
    └── CipherService.cs    -- wrapper service built on top of cipher.cs
```

### `ENCRYPTION_KEY` — Environment Variable

The encryption key is **never stored in config files**. It is read exclusively from the environment variable `ENCRYPTION_KEY`:

```csharp
// In Nova.Shared/Security/CipherService.cs
public sealed class CipherService : ICipherService
{
    private readonly string _key;

    public CipherService()
    {
        _key = Environment.GetEnvironmentVariable("ENCRYPTION_KEY")
               ?? throw new InvalidOperationException(
                   "ENCRYPTION_KEY environment variable is not set.");
    }

    public string Encrypt(string plainText)   => /* delegate to cipher.cs using _key */
    public string Decrypt(string cipherText)  => /* delegate to cipher.cs using _key */
}

public interface ICipherService
{
    string Encrypt(string plainText);
    string Decrypt(string cipherText);
}
```

The actual body of `Encrypt` and `Decrypt` must call the methods/functions found in the provided `cipher.cs`. Do not implement AES, RSA, or any algorithm directly in `CipherService` — it is a thin wrapper only.

### Encrypted Values in Configuration

All sensitive values in `appsettings.json` are stored encrypted. The convention for an encrypted value is a plain string — no prefix marker is required (the service always decrypts; it never receives plaintext in production).

During development/local only, plaintext values are acceptable — `CipherService.Decrypt` on an already-plaintext string must not throw; cipher.cs is expected to handle this gracefully or the developer sets the key to a known dev key.

### Where Decryption Is Applied

`IDbConnectionFactory` calls `ICipherService.Decrypt(connectionString)` before opening any connection. No other layer ever calls `ICipherService` directly — decryption is infrastructure-layer-only.

`TenantRegistry` stores connection strings as-encrypted; decryption happens inside `DbConnectionFactory.CreateForTenant()`, not in `TenantRegistry` itself.

`AppSettings.Jwt.SecretKey` is decrypted in `JwtSetupExtensions` before passing to `TokenValidationParameters`.

### Registration

```csharp
services.AddSingleton<ICipherService, CipherService>();
```

Singleton is correct — `CipherService` is stateless beyond the key read at construction.

## OpsSettings Hot-Reload Implementation Notes

- `OpsSettings` is bound via `IOptionsMonitor<OpsSettings>`
- A `OpsSettingsWatcher` hosted service subscribes to `OnChange`
- On change event:
  1. Validate new settings via `OpsSettingsValidator`
  2. If valid: update internal `CurrentOpsSettings` reference, log `[OpsSettings] Reloaded successfully`
  3. If invalid: log `[OpsSettings] Validation failed — retaining previous config: {errors}`, do not update reference
- All components that need ops settings inject `IOpsSettingsAccessor` (wrapper over the last known good instance)

## Constraints and Rules

1. No EF Core. No generic repositories.
2. No MediatR.
3. No AutoMapper.
4. No `DateTime.Now` anywhere — always `DateTime.UtcNow` or `DateTimeOffset.UtcNow`.
5. All queries must use parameterised inputs — no string interpolation in SQL.
6. `frz_ind = 0` filter on all domain table queries (not applicable to shell test queries against `pointer`).
7. English UK spelling throughout.
8. No `var` where the type is not obvious from the right-hand side.
9. XML doc `<summary>` on all public types and members in `Nova.Shared`.
10. Minimal API style in `Nova.Shell.Api` (not controller-based).
11. Do not generate migration scripts — schema is assumed to exist.
12. Do not implement caching beyond the settings model and interface definition.
13. Do not implement the outbox relay — only the `OutboxMessage` record is needed.
14. Never read `ENCRYPTION_KEY` from config files — always and only from `Environment.GetEnvironmentVariable("ENCRYPTION_KEY")`.
15. Never store plaintext connection strings, passwords, or JWT secrets in `appsettings.json` — always encrypted.
16. Decryption happens in infrastructure only (`DbConnectionFactory`, `JwtSetupExtensions`) — never in application or domain layers.
