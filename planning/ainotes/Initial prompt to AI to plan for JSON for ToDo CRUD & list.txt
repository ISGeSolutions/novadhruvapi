API Standards and Conventions:
**Base URL**
All endpoints are versioned and served under /api/v{n}/. The default version for each endpoint is defined in the static registry (src/services/apiNovaDhruvUxConfig.ts). Tenant-specific overrides may point individual endpoints to different versions.

**Wire Format — snake_case**
All JSON request and response bodies use snake_case key names on the wire (e.g., tenant_id, page_size, booking_no).

The frontend uses camelCase internally. A transform layer in apiClient (src/services/apiNovaDhruvUxConfig.ts) automatically converts:

Outgoing request bodies: camelCase → snake_case (via toSnakeCase)
Incoming response bodies: snake_case → camelCase (via toCamelCase)
Service files, components, and TypeScript interfaces never deal with snake_case directly. The examples below show the wire format (snake_case).

**Authentication**
All requests require a JWT Bearer token obtained via the auth endpoint.

**Obtain Token**
Method	POST
Path	/api/v1/auth/token
Body	{ "tenant_id": "T001", "shared_password": "..." }
Response	{ "token": "eyJhbG...", "expires_in": 3600 }
The shared_password is a per-tenant secret that authenticates the application instance to the API layer.

Once obtained, the token is sent on every subsequent request as:

Authorization: Bearer <token>
Note: The auth endpoint does not receive auto-injected context fields — it runs before context is established.
Note: Authorisation is handled by shared project.

**HTTP Method Convention**
All data-retrieval endpoints use POST (not GET). Request parameters are passed in the JSON body rather than as URL query strings. This keeps sensitive filters out of URLs, server logs, and browser history.

PATCH  (PATCH /api/v1/todos/{seq_no}) is retained for partial-update operations (e.g., saving records, saving booking client details).
In an update operation, do a get and if data is same as being updated, ignore (do not do the update)

- 200 — success. body has seq_no, updated_on
- 204 — no-op when submitted data is identical to DB 
- 404 — record not found
- 409 — concurrency conflict

Standard Context Fields (Auto-Injected)
Every request body routed through apiClient in src/services/apiNovaDhruvUxConfig.ts is automatically enriched with two context objects. These fields appear in every POST/PATCH request and must not be specified manually by service files.

**API Context (from tenant configuration)**
Field	Type	Description
tenant_id	string	Current tenant identifier
company_id	string	Current company identifier
branch_id	string	Current branch identifier
user_id	string	Current user identifier

**Client Context (captured on startup)**
Field	Type	Description
browser_locale	string	navigator.language (e.g., "en-GB")
browser_timezone	string	IANA timezone (e.g., "Europe/London")
ip_address	string | null	Client IP from ipify.org (fallback null). Server should also read X-Forwarded-For as a more reliable complement.
In the endpoint examples below, these 7 fields are shown with the comment // ← auto-injected to indicate they are added automatically by apiClient.

**Tenant-Specific Versioning**
The endpoint registry supports per-tenant overrides. When a tenant requires a different API version for a specific endpoint, it is configured in TENANT_OVERRIDES in src/services/apiNovaDhruvUxConfig.ts. The getEndpoint(key, tenantId?) function resolves the correct URL.

Example: If tenant T002 uses v2 for client-records, calls for that tenant resolve to /api/v2/client-records instead of /api/v1/client-records.

**Naming Conventions**

***API Endpoints***
Kebab-case, plural nouns: /api/v1/todos, /api/v1/booking-rows
Domain-prefixed: use todo-records not records, booking-rows not rows — avoids ambiguity as the API grows
Sub-resources via path nesting: /api/v1 bookings/{booking_no}/passengers/{passenger_id}
Version prefix: /api/v{n}/ — version is per-endpoint, resolved by getEndpoint()

***JSON Keys***
Wire format: snake_case for all JSON request and response bodies (e.g., tenant_id, page_size, booking_no)
