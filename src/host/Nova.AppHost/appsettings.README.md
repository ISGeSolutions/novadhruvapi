# AppHost configuration guide

## appsettings.json

Committed infrastructure toggles for local development. Change these to control which
containers Aspire starts when the AppHost launches.

### Infrastructure

| Key | Type | Default | Description |
|---|---|---|---|
| `UseRedis` | bool | `true` | Starts a Redis container (persistent). Required by any service that uses caching, distributed locking, or the Redis-backed outbox relay. When `false`, services must fall back to InMemory equivalents. |
| `UseRabbitMQ` | bool | `false` | Starts a RabbitMQ container (persistent) with the management plugin. Set to `true` when at least one tenant in any service's `appsettings.json` has `BrokerType: RabbitMq`. Management UI: http://localhost:15672 (guest/guest). |
| `UseSeq` | bool | `true` | Starts a Seq container (persistent) for unified structured log storage across all services. Seq UI: http://localhost:5341. Set to `false` only if you have an external log sink configured. |
| `UseMariaDb` | bool | `false` | Starts a MariaDB container (persistent) on port **3306**. Enable when testing MariaDB tenants locally. |
| `UsePostgres` | bool | `false` | Starts a Postgres container (persistent) on port **5432**, with pgAdmin accessible via the Aspire dashboard. Enable when testing Postgres tenants locally. |
| `UseMsSql` | bool | `false` | Starts a SQL Server container (persistent) on port **1433**. Only needed when a remotely hosted MSSQL server is not available — most developers leave this `false`. |

### DB container workflow

Only enable the containers you are actively testing. Each container runs independently — enabling one does not affect the others. All three use `WithLifetime(Persistent)` so data survives AppHost restarts.

**Typical use:**

```json
// Testing a MariaDB tenant — enable only MariaDB:
"Infrastructure": {
  "UseRedis": true,
  "UseRabbitMQ": false,
  "UseSeq": true,
  "UseMariaDb": true,
  "UsePostgres": false,
  "UseMsSql": false
}
```

**Connecting a tenant to the local container:**

The containers do not automatically inject connection strings into services. After starting a container, point the relevant tenant's `ConnectionString` in the service's `opsettings.json` to the container:

| Container | Host | Port | Example connection string |
|---|---|---|---|
| MariaDB | `localhost` | `3306` | `Server=localhost;Port=3306;Database=presets;Uid=root;Pwd=<pwd>;` |
| Postgres | `localhost` | `5432` | `Host=localhost;Port=5432;Database=presets;Username=postgres;Password=<pwd>;` |
| SQL Server | `localhost` | `1433` | `Server=localhost,1433;Database=presets;User Id=sa;Password=<pwd>;TrustServerCertificate=True;` |

The SA / root passwords are generated by Aspire and shown in the Aspire dashboard under the resource's **Details** panel when the container first starts. Copy them into `opsettings.json` (gitignored).

**Running migrations against the local container:**

Once the tenant connection string is in place, call the service's migration endpoint to initialise the schema:

```
POST /admin/migrations/run
Body: { "tenant_id": "YOUR_TENANT_ID", ... }
```

Alternatively run DbUp directly from the console endpoint if the service supports `--console` mode.

## appsettings.local.json

Machine-local secrets — **gitignored, never commit**.

Create this file by copying the placeholder below into
`src/host/Nova.AppHost/appsettings.local.json`:

```json
{
  "ENCRYPTION_KEY": "<your-encryption-key-here>"
}
```

| Key | Description |
|---|---|
| `ENCRYPTION_KEY` | Symmetric key used by `Nova.Cipher` to encrypt/decrypt connection strings and other secrets in each service's `appsettings.json`. Must match the key that was used when those values were encrypted. |

The AppHost reads this value and passes it to every service via `.WithEnvironment("ENCRYPTION_KEY", ...)`.
